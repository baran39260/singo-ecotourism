# سند پروژه سینگو (Singo) — محصول تمپلیت وب‌سایت بومگردی و اقامتگاه

> **نسخه:** ۰.۳ (بازنگری با محوریت «محصول قابل فروش»)
> **تاریخ:** ۱۴۰۵/۰۲/۰۴ — 2026-04-24
> **وضعیت:** تثبیت چشم‌انداز محصول، آماده شروع ساخت دمو

---

## ۱. معرفی پروژه

**سینگو (Singo)** یک **محصول تمپلیت حرفه‌ای برای وب‌سایت اقامتگاه‌ها و بومگردهای ایران** است. هدف ساخت **یک سایت نمونه (Showcase) درجه‌یک** است که هم به‌عنوان ویترین فروش کار کند و هم به‌عنوان پایه‌ای که آن را به **صاحبان اقامتگاه‌ها، هتل‌های کوچک و متوسط، و بومگردها** بفروشیم تا هرکدام **سایت اختصاصی خودشان** را داشته باشند.

نام «سینگو» در گویش قشمی به معنای **خرچنگ** است و نماد برند شرکت ماست (توسعه‌دهنده محصول)، نه نام سایت مشتری.

### تفاوت بنیادی با نسخه قبلی (۰.۲)

نسخه ۰.۲ یک پلتفرم چند-مستاجری (SaaS) پیش‌بینی کرده بود. **این تصمیم لغو شد.** مدل محصول جدید:

| مشخصه                      | نسخه قبل (SaaS)         | نسخه جدید (محصول قابل فروش)                |
| -------------------------- | ----------------------- | ------------------------------------------ |
| تعداد دیپلوی               | یک دیپلوی، همه مشتری‌ها | یک دیپلوی مستقل برای هر مشتری              |
| دیتابیس                    | مشترک + `tenantId`      | مستقل برای هر مشتری                        |
| دامنه                      | زیردامنه یا کاستوم      | دامنه کاملاً اختصاصی مشتری                 |
| مدل درآمد                  | کمیسیون/اشتراک          | **لایسنس یک‌باره + راه‌اندازی + پشتیبانی** |
| ریسک نشت داده بین مشتری‌ها | بحرانی                  | ندارد (ایزوله فیزیکی)                      |
| به‌روزرسانی                | خودکار برای همه         | بسته‌های نسخه‌دار + ابزار مهاجرت           |

این تغییر معماری، بسیاری از پیچیدگی‌های فنی نسخه قبلی را (Tenant Resolution در Middleware، Row-Level Security، Prisma Client Extensions برای ایزوله‌سازی) **حذف می‌کند** و تمرکز را روی **کیفیت محصول و سهولت سفارشی‌سازی** می‌گذارد.

### چرا این مدل؟ (ارزیابی بازار)

- بازخورد کارشناسان صنعت: صاحب بومگرد کوچک حاضر به پرداخت اشتراک ماهانه ثابت نیست. اما **برای یک سایت حرفه‌ای با مالکیت کامل، حاضر به پرداخت یک‌باره ۳ تا ۱۵ میلیون تومان هست** (بسته به سطح سفارشی‌سازی).
- رقبا (جاباما، شب، اقامت۲۴) **کانال فروش** می‌فروشند، نه **سایت مستقل**. فاصله این بازار خالی است.
- سایت اختصاصی برای اقامتگاه = **کمیسیون صفر** روی رزرو مستقیم + برند مستقل + SEO خودش.

### اهداف محصول

1. **ساخت یک سایت دمو (Showcase) درجه‌یک** که کیفیت و قابلیت‌های ما را نشان دهد
2. ساخت **یک تمپلیت قابل تکرار** که تیم ما بتواند برای هر مشتری در **کمتر از ۲ هفته** مستقر کند
3. سفارشی‌سازی ظاهری (رنگ، لوگو، محتوا) فقط با **یک فایل کانفیگ + پنل ادمین**، بدون دست‌کاری کد
4. رعایت تمام **الزامات قانونی ایران** (سامانه مسافر پلیس اماکن، مالیات ۹٪ گردشگری، سامانه مؤدیان) به صورت built-in
5. **Instant Booking** و پشتیبانی کامل فارسی/RTL/تقویم شمسی
6. کیفیت فنی: Lighthouse ≥ ۹۰، Core Web Vitals عالی، SEO پخته

### مخاطب‌های محصول

۱. **اقامتگاه‌های سنتی و بومگردی** (قشم، کاشان، یزد، اصفهان، کردستان، گیلان)
۲. **هتل‌های کوچک و متوسط بوتیک** (۵-۳۰ اتاقه)
۳. **مجموعه‌های چند-اقامتگاهی** یک مالک (می‌توانند چند سایت بخرند با تخفیف)
۴. **آژانس‌های کوچک گردشگری** که پکیج می‌فروشند

## ۲. استراتژی محصول و مدل کسب‌وکار

### ۲.۱ سبد محصول (Product Tiers)

| طرح                          | قیمت (تومان)           | شامل                                                                                                  |
| ---------------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------- |
| **Starter**                  | ۳-۵ میلیون (یک‌باره)   | سایت با دامنه، نصب روی هاست مشتری، تم از لیست ما، پنل ادمین، پشتیبانی ۳ ماه                           |
| **Professional**             | ۸-۱۲ میلیون (یک‌باره)  | همه Starter + عکاسی حرفه‌ای + ورود اولیه محتوا + آموزش + پشتیبانی ۶ ماه                               |
| **Premium**                  | ۱۵-۲۵ میلیون (یک‌باره) | همه Professional + سفارشی‌سازی ظاهری + اتصال درگاه پرداخت اختصاصی + یکپارچگی با سیستم‌های داخلی مشتری |
| **Managed Hosting** (افزونه) | ۵۰۰ هزار/ماه           | هاستینگ + مانیتورینگ + پشتیبان‌گیری + آپدیت خودکار نسخه‌ها                                            |
| **پشتیبانی تمدید**           | ۱ میلیون/سال           | باگ‌فیکس، نسخه جدید، پاسخ تیکت                                                                        |

> اعداد اولیه پیشنهادی هستند و باید با تست بازار تنظیم شوند.

### ۲.۲ مدل به‌روزرسانی (موضوع حیاتی)

مشتری نسخه `v1.0` را می‌خرد. اگر `v1.1` باگ امنیتی برطرف می‌کند، چطور به او می‌رسد؟

**سه سطح:**

1. **Self-hosted رایگان:** مشتری فقط باگ‌فیکس‌های امنیتی بحرانی را مجانی می‌گیرد (patch zip + دستورالعمل)
2. **اشتراک پشتیبانی:** همه نسخه‌ها + باگ‌فیکس + ارتقای وابستگی‌ها را می‌گیرد
3. **Managed Hosting:** ما خودمان آپدیت را روی دیپلوی مشتری اعمال می‌کنیم

**اصل مهندسی مهم:** سفارشی‌سازی فقط از طریق **config file + پنل ادمین + theme overrides** مجاز است. هر مشتری که کد هسته را دست‌کاری کند، از پوشش آپدیت خارج می‌شود. این قاعده باید از روز اول در ساختار ریپو اعمال شود (دایرکتوری `overrides/` مجاز، `src/core/` غیرمجاز).

### ۲.۳ خدمات جانبی (حاشیه سود بالا)

- عکاسی حرفه‌ای اقامتگاه (۲-۴ میلیون)
- ورود محتوا و تولید بلاگ اولیه (۱-۲ میلیون)
- اتصال به سامانه‌های قانونی (مسافر پلیس اماکن، مؤدیان) — ۵۰۰ هزار تومان راه‌اندازی
- آموزش صاحب/کارمند اقامتگاه
- مدیریت اینستاگرام و تولید محتوا (ماهانه)

## ۳. معماری فنی: یک دیپلوی مستقل برای هر مشتری

### ۳.۱ مدل انتخاب‌شده

**تمپلیت تک-مستاجری (Single-tenant) + استراتژی نسخه‌بندی ریپو.** هر مشتری یک ریپو (یا branch جداگانه) + یک دیتابیس + یک دیپلوی مستقل دارد.

```
singo-template (ریپو اصلی ما)
  ├── v1.0 (تگ)
  ├── v1.1 (تگ)
  └── v1.2 (تگ)
         │
         ├── clone + customize → delgosha-site  (ریپو مشتری ۱)
         ├── clone + customize → hara-site      (ریپو مشتری ۲)
         └── clone + customize → naz-hotel      (ریپو مشتری ۳)
```

### ۳.۲ چرا نه Monorepo؟

برای تیم دو نفره در آغاز، monorepo با پکیج core مشترک هزینه DX بالایی دارد (pnpm workspaces + Turborepo + مدیریت نسخه داخلی). اگر در آینده تعداد مشتری‌های فعال زیر پوشش Managed Hosting به بیش از ۱۰ رسید، **می‌توان به monorepo مهاجرت کرد** (مدل زیر):

```
singo-platform/
  ├── packages/core          # منطق و کامپوننت‌های مشترک
  ├── packages/ui
  └── apps/
      ├── demo               # سایت دمو
      ├── delgosha
      └── hara
```

این مهاجرت «در صورت نیاز» است، نه از روز اول.

### ۳.۳ ایزوله‌سازی

- هر مشتری دیتابیس جداگانه PostgreSQL
- هر مشتری bucket/پوشه S3 جداگانه
- هر مشتری domain + SSL اختصاصی
- هر مشتری Auth.js secret اختصاصی
- **هیچ داده‌ای بین مشتری‌ها به اشتراک گذاشته نمی‌شود**

> پیچیدگی‌های نسخه قبلی (Middleware host resolution، Row-Level Security، Prisma Client Extensions برای اعمال `tenantId`) **حذف شدند**. اسکیمای دیتابیس هیچ ستون `tenantId` ندارد.

### ۴.۲ محتوا از پنل ادمین

داده‌های پویا (اقامتگاه‌ها، پکیج‌ها، گالری، بلاگ، رزروها) از پنل ادمین مدیریت می‌شوند، نه فایل کانفیگ.

### ۴.۳ Theme Overrides (برای پلن Premium)

## ۵. اصول طراحی UI

### یوزر اکسپرینس

- **موبایل-اول** (≥۷۵٪ ترافیک سفر از موبایل)

- **RTL و فارسی** پیش‌فرض؛ چندزبانه (عربی/انگلیسی) به‌صورت feature flag

### نکات معماری Next.js 15

- **PPR (Partial Prerendering):** hero و CTA استاتیک، لیست اقامتگاه‌ها dynamic
- **`generateMetadata`** برای متادیتای SEO
- **`app/sitemap.ts` و `app/robots.ts`** پویا
- **`opengraph-image.tsx`** با `ImageResponse` و رنگ برند
- **LodgingBusiness schema.org** (JSON-LD) در صفحه اقامتگاه
- **`after()`** برای ارسال پیامک/ایمیل بعد از پاسخ به کاربر
- **Auth.js v5 split config:** `auth.config.ts` edge-safe برای middleware، `auth.ts` با Prisma adapter در Node

## ۷. ویژگی‌های اصلی

### ۷.۱ صفحات عمومی (برای سایت مشتری)

- **صفحه اصلی:** هرو، اقامتگاه‌های ویژه، جاذبه‌های نزدیک، تستیمونیال، CTA رزرو
- **لیست اقامتگاه‌ها/اتاق‌ها:** فیلتر تاریخ/قیمت/ظرفیت
- **جزئیات اقامتگاه:** گالری، توضیحات، امکانات، نقشه، **تقویم دسترسی، دکمه رزرو فوری**
- **جاذبه‌های نزدیک:** محلی، قابل ویرایش از پنل
- **بلاگ** (اختیاری با feature flag)
- **گالری، درباره ما، تماس، قوانین و سیاست لغو**
- **FAQ** (پرتکرار در صنعت)

### ۷.۲ کاربری

- ثبت‌نام/ورود (موبایل + OTP، ایمیل + پسورد اختیاری)
- پروفایل، تاریخچه رزرو، علاقه‌مندی، نظرات

### ۷.۳ رزرو (مدل جدید: Instant Booking اصلی + بافر اختیاری)

**تغییر بنیادی نسبت به نسخه قبلی.** بر اساس بازخورد کارشناس صنعت، مدل Request-to-Book به‌تنهایی **نرخ تبدیل را ۴۰-۶۰٪ می‌کُشد** چون مشتری ایرانی به Instant Booking جاباما عادت کرده.

**مدل Hybrid پیش‌فرض:**

1. ۷۰٪ ظرفیت **Instant Booking** مستقیم (قابل تنظیم در config)
2. ۳۰٪ بافر برای کانال‌های بیرونی (جاباما، شب)
3. در صورت رزرو Instant → قفل ۱۵ دقیقه‌ای ظرفیت هنگام پرداخت
4. **deadline تأیید ۳۰ دقیقه (نه ۶ ساعت)** برای سناریوهای خاص که owner باید تأیید کند؛ بعد از آن اتومات تأیید مگر رد شود
5. **بافر پویا** بر اساس فصل (اوج/کساد) قابل تنظیم
6. تقویم مسدودی دستی برای ست کردن رزروهای جاباما/شب

### ۷.۴ سیاست لغو (Cancellation Policy)

سه سطح استاندارد به انتخاب مشتری:

- **Flexible:** لغو تا ۲۴ ساعت قبل، استرداد کامل
- **Moderate:** لغو تا ۷۲ ساعت قبل، استرداد کامل؛ کمتر از آن ۵۰٪
- **Strict:** استرداد فقط در لغو بیش از ۷ روز قبل

### ۷.۵ پنل صاحب اقامتگاه (Owner Panel) — **واتساپ-محور**

بازخورد صنعت: صاحب اقامتگاه ۴۰+ ساله معمولاً به پنل وب سر نمی‌زند. **ارتباط اصلی باید از واتساپ باشد.**

- رزرو جدید → **اعلان واتساپ فوری** با دکمه تأیید/رد (لینک deep-link به پنل)
- داشبورد ساده: رزروهای امروز، فردا، هفته آینده، درآمد ماه
- مدیریت اتاق‌ها، قیمت‌گذاری، تقویم مسدودی
- مدیریت گالری، بلاگ
- **چک‌لیست نظافت** بین دو میهمان + تخصیص به نیروی خدمات
- تنظیمات برند (رنگ، لوگو، اطلاعات تماس) — فقط در پلن Professional+
- گزارش مالی

### ۷.۶ ماژول انطباق قانونی ایران (Built-in، غیرقابل حذف)

این ماژول در نسخه قبلی غایب بود و بازخورد کارشناس صنعت حیاتی دانست:

1. **ثبت اجباری کد ملی میهمان** در فرم رزرو + اعتبارسنجی الگوریتم کد ملی
2. **ارسال خودکار به سامانه مسافر پلیس اماکن** هنگام Check-in
3. **محاسبه مالیات ارزش افزوده ۹٪ گردشگری** و تفکیک بهای اقامت/غذا/خدمات جانبی
4. **صدور فاکتور الکترونیکی سامانه مؤدیان** (اجباری از ۱۴۰۳) — ارسال و دریافت شناسه یکتا
5. **گزارش ورود/خروج** به سازمان میراث/منطقه آزاد (برای اقامتگاه‌های دارای مجوز)
6. مدیریت ودیعه (Security Deposit) اختیاری

> اسکیمای دیتابیس و APIها باید از روز اول این الزامات را ببینند تا بعداً refactor سنگین نشود.

## ۹. نقشه راه (بازنگری‌شده)

تمرکز فاز اول: **ساخت سایت دمو با کیفیت بالا** برای ویترین فروش. محصول‌سازی و فروش در فاز دوم.

| فاز                                  | هفته  | خروجی‌ها                                                                                           |
| ------------------------------------ | ----- | -------------------------------------------------------------------------------------------------- |
| ۱. طراحی و پایه                      | ۱-۲   | طراحی Figma کامل، انتخاب برند دمو (فرضی یا پایلوت واقعی)، ساختار Next.js 15 + Tailwind v4 + Prisma |
| ۲. احراز هویت + نقش‌ها               | ۳     | Auth.js v5 split config، ثبت‌نام OTP، نقش‌های user/owner/admin                                     |
| ۳. CRUD محتوا + پنل Owner            | ۴-۵   | مدیریت اتاق، گالری، تقویم، تنظیمات برند؛ **اعلان واتساپ**                                          |
| ۴. صفحات عمومی                       | ۶-۷   | هوم، لیست، جزئیات، جاذبه‌ها، بلاگ، درباره، تماس؛ PPR و generateMetadata                            |
| ۵. سیستم رزرو                        | ۸-۹   | Instant Booking + hybrid، تقویم مسدودی، state machine، قفل ظرفیت، سیاست لغو                        |
| ۶. درگاه پرداخت + انطباق قانونی      | ۱۰-۱۱ | زرین‌پال، محاسبه مالیات ۹٪، سامانه مسافر، سامانه مؤدیان                                            |

## ۱۱. فرآیند فروش و راه‌اندازی مشتری جدید

بازخورد کارشناس صنعت: راه‌اندازی یک مشتری **۲-۴ هفته** زمان می‌برد، نه «کمتر از یک روز».

1. **تماس اولیه** (معمولاً از طریق معرفی یا اینستاگرام)
2. **جلسه مشاوره** (حضوری/آنلاین) — نیازسنجی، انتخاب طرح
3. **بازدید حضوری + عکاسی حرفه‌ای** (برای طرح Professional+)
4. **جمع‌آوری مدارک:** مجوز، کد ملی مالک، شماره شبا، کد پستی، اطلاعات مالیاتی
5. **امضای قرارداد و پیش‌پرداخت**
6. **تنظیم `site.config.ts`** توسط تیم سینگو
7. **تهیه دامنه + SSL + دیپلوی روی لیارا/آروان**
8. **ورود اولیه محتوا** (اتاق‌ها، قیمت‌ها، گالری) — توسط تیم ما
9. **آموزش پنل** به صاحب (۲ ساعت حضوری یا ویدیو)
10. **رزرو آزمایشی داخلی**
11. **لایو شدن + تحویل + شروع دوره پشتیبانی**

**از تماس اولیه تا لایو: ۲-۴ هفته.** این واقع‌بینی باید در ارائه به مشتری روشن باشد.

### محصول و کسب‌وکار

- [ ] قیمت‌گذاری نهایی سه طرح (پیشنهاد اولیه: ۵/۱۰/۲۰ میلیون)

### تیم

- [ ] تقسیم کار (فرانت/بک یا فول‌استک بر اساس فیچر)
- [ ] فریلنسرهای جانبی (طراح گرافیک، عکاس، تولیدکننده محتوا)

## ۱۴. معیارهای موفقیت

### KPIهای محصول (کیفیت فنی)

- Lighthouse Performance ≥ ۹۰ در موبایل
- Lighthouse SEO ≥ ۹۵
- Core Web Vitals: همه در محدوده Good
- Time to First Booking در سایت دمو ≤ ۳ کلیک از landing

### KPIهای کسب‌وکار (۶ ماه پس از لانچ)

- **۳-۵ مشتری پرداختی** (محافظه‌کارانه؛ کارشناس صنعت ۱۰-۱۵ را هدف مناسب دانست ولی برای MVP ریالیستیک ۳-۵)
- درآمد ۳۰-۶۰ میلیون تومان فقط از لایسنس
- ۲ مشتری در پلن Professional+
- حداقل ۱ مشتری در Managed Hosting

### KPIهای صنعتی در سایت مشتری‌ها (اندازه‌گیری پس از لایو هر مشتری)

- **Occupancy Rate** (نرخ اشغال): هدف ≥ ۶۰٪ در فصل اوج
- **ADR** (Average Daily Rate)
- **RevPAR** = Occupancy × ADR — شاخص اصلی سودآوری
- **Direct Booking Share:** سهم رزرو مستقیم از سایت vs. کانال‌های بیرونی — هدف ۴۰٪ در ماه ۶
- **Cancellation Rate** و **No-show Rate**
- **Repeat Guest Rate**
