# AGENTS.md — راهنمای AI برای اوپن‌اسپک سینگو

این فایل برای دستیارهای AI (Claude، Cursor، Copilot، …) است که در این مخزن کار می‌کنند. قبل از هر تغییر در `openspec/` یا `src/`، این سند را بخوانید.

## ترتیب خواندن

۱. `openspec/project.md` — اصول بنیادی پروژه
۲. `openspec/README.md` — ساختار و نقشه پروپوزال‌ها
۳. پروپوزال فعلی در `openspec/changes/<id>/`
۴. `docs/PROJECT.md` — سند جامع محصول (اگر context بیشتری لازم است)

## اصول غیرقابل نقض (Non-negotiable)

این‌ها در `project.md` به‌تفصیل آمده. خلاصه:

1. **تک-مستاجری است.** هیچ `tenantId` در اسکیما ننویس.
2. **مرز core/overrides:** `src/core/` هرگز از `@/overrides` import نمی‌کند. این قاعده با ESLint rule اجباری است.
3. **انطباق قانونی ایران:** هر فیچر رزرو باید کد ملی، مالیات ۹٪ VAT و مؤدیان را ببیند.
4. **رزرو پیش‌فرض Instant.** نه Request-to-Book.
5. **اعلان owner از واتساپ می‌آید**، نه فقط پنل وب.
6. **RTL فارسی پیش‌فرض.** چندزبانه با feature flag.
7. **هر متن UI از i18n layer می‌آید، نه hardcode.**
8. **هیچ secret در کد.** فقط env.

## جریان کار AI

### وقتی کاربر می‌گوید «فیچر X را اضافه کن»

۱. بررسی کن آیا پروپوزالی برای این فیچر وجود دارد در `openspec/changes/`؟

- اگر هست → فقط taskهای آن را اجرا کن
- اگر نیست → **اول پروپوزال بنویس**، به کاربر بده، منتظر تأیید باش

۲. پروپوزال شامل این ۴ فایل است:

- `proposal.md` — Why / What / Impact / Acceptance
- `tasks.md` — چک‌لیست عملی
- `design.md` — تصمیمات فنی غیر-پیش‌پا‌افتاده (اختیاری برای تغییرات ساده)
- `specs/<capability>/spec.md` — requirements با scenarioهای Gherkin

۳. حین پیاده‌سازی:

- هر task که تمام شد، در `tasks.md` تیک بزن (`- [ ]` → `- [x]`)
- اگر تصمیم فنی جدیدی گرفتی که در `design.md` نبود، به `design.md` اضافه کن
- اگر spec تغییر کرد، delta را در همان پروپوزال آپدیت کن

۴. وقتی همه taskها تیک خورد:

- Acceptance Criteria را چک کن
- spec deltas را به `openspec/specs/<capability>/spec.md` ادغام کن
- پروپوزال را به `openspec/changes/archive/` منتقل کن

### وقتی کاربر می‌گوید «این را بفهم/توضیح بده»

۱. اگر درباره وضعیت فعلی است → `openspec/specs/` را بخوان
۲. اگر درباره تصمیم گرفته‌شده است → `openspec/changes/<id>/design.md` را بخوان
۳. اگر درباره «چرا این‌طور ساخته شده» است → `design.md` و commit history

## قواعد نوشتن spec

هر Requirement این قالب را دارد:

```markdown
### Requirement: <توصیف یک‌جمله‌ای>

<توضیح متن آزاد در ۱-۳ پاراگراف>

#### Scenario: <نام سناریو>

- **GIVEN** <حالت اولیه>
- **WHEN** <اقدام>
- **THEN** <نتیجه مورد انتظار>
- **AND** <نتیجه اضافی>
```

Requirementها در فایل delta با یکی از سه عنوان گروه‌بندی می‌شوند:

- `## ADDED Requirements` — جدید
- `## MODIFIED Requirements` — تغییر در requirement موجود
- `## REMOVED Requirements` — حذف

## قواعد نوشتن task

- هر task **عمل‌گرا** و **قابل چک کردن** باشد
- شماره‌گذاری دو-سطحی: `1.1`, `1.2`, ...
- جمله‌ای که بتوان بر اساس آن PR نوشت
- اگر task بیشتر از ۴ ساعت کار دارد، به دو task بشکن

## کارهایی که AI نباید انجام دهد

- **بدون پروپوزال تأییدشده** شروع به کدنویسی روی فیچر جدید نکن
- فایل‌های پروپوزال تأییدشده را **بدون اطلاع کاربر** تغییر نده
- اصول `project.md` را در پروپوزال جدید **نقض نکن** (اگر لازم است، اول `project.md` را آپدیت کن)
- بین پروپوزال‌ها **پرش نکن** (پروپوزال ۳ بدون تکمیل ۲ شروع نشود، مگر کاربر صریحاً بگوید)
- `src/core/` را بدون نیاز روشن **تغییر نده**

## commit conventions

- هر commit به یک task تیک‌خورده اشاره کند: `[1-foundation] task 5.3: implement DB session`
- commit پیام فارسی قابل قبول است ولی شناسه task در براکت انگلیسی
